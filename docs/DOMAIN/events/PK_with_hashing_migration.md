# PK Hashing Migration ŌĆö Slugs to Hashids

**Status:** Design only, no code written.

This document describes the migration from slug-based URL addressing (stored `CharField` on the model) to hashid-encoded integer PKs exposed at the URL boundary. The control layer and database are unaffected ŌĆö only the URL pattern and the presentation-layer lookup change.

---

## Why

Slugs are a stored field: they require a separate column, a uniqueness constraint, a generation function (`generate_slug()`), and a collision-retry loop in `save()`. They exist only to give URLs a non-guessable identifier. None of this is load-bearing ŌĆö the integer PK already uniquely identifies every row.

Hashids encode an integer PK deterministically using a secret salt. The encoded string is non-guessable without the salt, human-readable in URLs, configurable in length, and requires zero database storage. Decoding at the entrypoint yields the integer PK directly. No slug field, no uniqueness logic, no stored state.

---

## What changes

### Removed from models

| Model | Removed | Also removed |
|---|---|---|
| `Event` | `slug = CharField(...)` | `_unique_slug()`, slug assignment in `save()` |
| `EventComment` | `slug = CharField(...)` | `_unique_slug()`, slug assignment in `save()` |

### Removed from utilities

`generate_slug()` in `app/events/utils.py` is removed. `generate_uuid7()` stays ŌĆö it is used by `EventFile` and `CommentAttachment`, which remain UUID7.

### What stays unchanged

`EventFile` and `CommentAttachment` keep their UUID7 primary keys. Their URLs use the raw UUID directly ŌĆö no hashid encoding. Hashids are only applied to integer PK tables (`Event`, `EventComment`).

## The hashids utility

**Location:** `app/utils/hashids.py`

```python
from hashids import Hashids
from django.conf import settings

def _hashids() -> Hashids:
    return Hashids(salt=settings.HASHIDS_SALT, min_length=8)

def encode_id(pk: int) -> str:
    """Encode an integer PK to an 8-character hashid string."""
    return _hashids().encode(pk)

def decode_hash(hash_str: str) -> int | None:
    """Decode a hashid string to an integer PK. Returns None if invalid."""
    result = _hashids().decode(hash_str)
    return result[0] if result else None
```

Characteristics:
- Output is always at least 8 characters (`min_length=8`).
- The same PK + same salt always produces the same hashid (deterministic).
- An invalid or tampered hash decodes to `None` ŌĆö treat as 404.
- The salt is secret; without it the encoding cannot be reversed.

## Environment and settings

### `.env` (generated by `generate_env.py`)

A new key is added:

```
# --- Hashids (URL-safe PK encoding) ---
HASHIDS_SALT=<auto-generated 32-character random string>
```

### `generate_env.py` changes

`EnvGenerator.build_content()` generates and includes `HASHIDS_SALT`:

```python
hashids_salt = secrets.token_urlsafe(32)   # ~43 characters, cryptographically random

# Added to the content string:
# --- Hashids ---
HASHIDS_SALT={hashids_salt}
```

Dev mode uses a fixed, documented insecure salt:

```
HASHIDS_SALT=dev-insecure-hashids-salt-do-not-use-in-production
```

### `config/settings.py`

```python
HASHIDS_SALT = env("HASHIDS_SALT")
```

## URL patterns

Slugs in URL patterns become generic `<str:hash>` (or `<str:event_hash>` / `<str:comment_hash>` when two appear in the same path):

**Before:**
```python
path("<str:slug>/", event_detail, name="event_detail"),
path("<str:slug>/edit/", event_edit, name="event_edit"),
path("<str:event_slug>/comments/<str:comment_slug>/edit/", comment_edit, ...),
```

**After:**
```python
path("<str:hash>/", event_detail, name="event_detail"),
path("<str:hash>/edit/", event_edit, name="event_edit"),
path("<str:event_hash>/comments/<str:comment_hash>/edit/", comment_edit, ...),
```

## Entrypoint lookup pattern

Every entrypoint that previously called `get_event_by_slug(slug)` now decodes the hashid and fetches by integer PK.

**Before:**
```python
event = get_event_by_slug(slug)
```

**After:**
```python
from app.utils.hashids import decode_hash

event_id = decode_hash(hash)
event = get_object_or_404(Event.objects.active(), pk=event_id)
```

`decode_hash` returning `None` (invalid or tampered hash) causes `get_object_or_404` to raise 404 ŌĆö no special error handling needed, since `get_object_or_404(Event, pk=None)` will 404 naturally. If a cleaner 404 is preferred, add an explicit guard:

```python
event_id = decode_hash(hash)
if event_id is None:
    raise Http404
event = get_object_or_404(Event.objects.active(), pk=event_id)
```

The same pattern applies to comment lookups.

## Template URL generation

Templates that build URLs for events and comments must encode the integer PK. Register a template tag or use a context variable.

**Option A ŌĆö template tag:**
```
{% load hashid_tags %}
<a href="{% url 'event_detail' hash=event.pk|encode_id %}">{{ event.title }}</a>
```

**Option B ŌĆö pre-encoded in context (simpler):**
```python
# In the entrypoint:
from app.utils.hashids import encode_id
context["event_hash"] = encode_id(event.pk)
```

```html
<a href="{% url 'event_detail' hash=event_hash %}">{{ event.title }}</a>
```

Option B is simpler for most views. Option A is useful when rendering lists of events where each row needs its own hash.

**Structs and `to_dict()`:** The `BaseEventStruct.to_dict()` output should include the encoded hashid alongside the integer PK, so callers receive a ready-to-use URL identifier:

```
BaseEventStruct.to_dict() ŌåÆ {
  "event_id": int,
  "event_hash": str,   ŌåÉ encode_id(event.pk)
  ...
}
```

## Forms with hidden ID fields

Any form that POSTs to a resource-specific URL should include the integer PK as a hidden field so the action does not depend solely on the URL hashid. This prevents stale references if context changes mid-session.

```html
<input type="hidden" name="event_id" value="{{ event.id }}">
```

The entrypoint reads `event_id` from POST data as the authoritative identifier, decoding the URL hash only as a fallback or for initial resolution.
