Data domains and scope

A data domain is the atomic unit of row-level access control. Every database row that contains sensitive data is tagged with a domain. Users are assigned domains, and only see rows in their assigned domains.

The Golden Rule: A user can see a row if and only if the row's domain is in the user's assigned domains. Nothing else grants access at the row level.

The domain primitive

Every sensitive data row carries exactly one domain foreign key:

  • An Asset row has `domain_id = 5` (e.g., "Facility 1 Transportation")
  • An Event row has `domain_id = 7` (e.g., "Headquarters Operations")

A user is assigned zero or more domains:

  • Jane is assigned domains 5 and 7
  • On every query, only rows with `domain_id in [5, 7]` are returned

This is the sole mechanism for row-level access. No other system — not roles, not organizations, not divisions — grants row visibility.

How users are assigned domains

Domains are assigned to users in two ways:

Via domain template (bulk):
A named template bundles related domains (e.g., "Facility 1 Transportation" = transportation domain + supply domain). Assign the template → user gets all its domains.
Via explicit assignment (ad-hoc):
Directly assign a single domain to a user outside any template.

The organizational hierarchy

The system has a three-level hierarchy of organization:

Division (e.g., "Transportation")
  └─ Organization (e.g., "Facility 1")
       └─ Domain (e.g., "Facility 1 Transportation")

Important: This hierarchy is for navigation and administration only. It does not grant access.

Example: Assigning a user to the "Facility 1" organization does not automatically let them see Facility 1's rows. They must be explicitly assigned the domains under that organization.

The hierarchy exists to help operators find and organize domains, not to control access.

Cross-organization and cross-division grants

It is expected and allowed for a user to be granted domains outside their organization or division:

  • An IT technician from the IT organization needs read access to a Facilities domain to troubleshoot kiosks
  • A shared-service auditor needs domains across every division
  • A contractor is "on loan" from one organization to another

Rather than prohibiting these grants, the system allows them and makes them visible so operators can audit unusual access.

The admin warning system

When viewing a user's domain assignments, the system highlights grants that fall outside the user's assigned organization or division:

Color Condition
Red The domain belongs to an organization the user is not assigned to.
Yellow The domain's organization matches, but the division differs (or vice versa).
Default The domain belongs to one of the user's assigned organization/division pairs.

Note: The warning is an audit aid. The grant still works regardless of color.
