Roles and permission groups

A role is a job title (e.g., "Technician", "IT Manager") that bundles together permission groups. Roles let you assign multiple permissions at once and tell a story about what a person is responsible for.

Permission groups

A permission group is an atomic bundle of permissions tied to a specific action or feature:

  • Asset Lifecycle — permissions to add, change, delete, view assets
  • Supply Management — permissions to manage supply requests
  • Audit Report Generation — permissions to generate compliance reports

Permission groups are not organizational teams or departments — they represent specific job functions.

Roles

A role is a real-world job profile that bundles multiple permission groups:

  • Technician → Asset Lifecycle, Equipment Tracking
  • IT Manager → User Administration, System Configuration
  • Facilities Auditor → Audit Reports, Facility Access

Assigning a role to a user means: "This person should be able to do everything this role requires." Instead of manually assigning 10 permission groups to 20 users (200 clicks), assign one role to 20 users (20 clicks).

Multiple roles per user

A user can hold multiple roles at the same time. Their effective permissions are the union of all roles:

Example:

User: Jane
├─ Role: Technician
│  └─ Groups: [Asset Lifecycle, Equipment Tracking]
└─ Role: Facilities Auditor
   └─ Groups: [Audit Reports, Facility Access]

Effective groups: {Asset Lifecycle, Equipment Tracking, Audit Reports, Facility Access}

Role inheritance (one level)

Roles can form a parent-child relationship to represent specialization:

  • Base role: "Technician" (has no parent)
  • Specialized role: "Documentation Technician" (parent: Technician) adds documentation responsibilities on top

The child role automatically includes all permission groups from its parent. When a user holds the specialized role, they get both parent and child groups (no overlap).

Constraint: Role hierarchy is only one level deep. A specialized role cannot itself be a parent.

Relationship types and notes

Each role assignment includes a relationship type that contextualizes the assignment:

  • Primary — the user's main job title
  • Specialty — a secondary specialization
  • Side job — occasional duties outside the main role
  • For learning/fun — training or temporary coverage

Additionally, each assignment includes a notes field that explains why the user holds this role. Examples:

  • "Primary role — hired as field technician."
  • "Specialty added 2026-03-15 for Q1 documentation project; sunset 2026-09-01."
  • "Side job: supply coverage during Sarah's leave (until 2026-05-30)."
Important: Roles grant capabilities only. They have zero effect on data scope. A user with the "Technician" role still sees no rows until they are assigned domains. Learn about data domains →
