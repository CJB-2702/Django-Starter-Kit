How access control works

This system controls access through two independent gates. Both must pass for a user to perform an action or see data.

The Two Gates: A user must clear both a capability gate (what actions they may perform) and a scope gate (what rows they may see). Neither implies the other.

Gate 1 — Capability (What you can do)

Question: Does this user have permission to perform this action at all?

Answer: Django checks the user's permission groups.

  1. User holds a role (e.g., "Technician")
  2. Each role bundles multiple permission groups (e.g., "Asset Lifecycle", "Equipment Tracking")
  3. Each permission group grants specific actions (add/change/view/delete a model)
  4. Django checks: "Can this user add an asset?" Yes → proceed. No → denied.

Related: Roles and permission groups

Gate 2 — Scope (What you can see)

Question: Does this user have access to the data domains this row belongs to?

Answer: Every row carries a domain. Every user is assigned domains.

  1. Every database row lives in a data domain (e.g., "Facility 1 Transportation")
  2. User is assigned one or more domains (directly or via template)
  3. On every query, only rows matching the user's domain set are returned
  4. Check: "Is row.domain in my assigned domains?" Yes → visible. No → filtered out.

Related: Data domains and scope · Domain templates

Both gates must pass

Example: Jane is a Technician and has the "Asset Lifecycle" permission group, so she can add assets. But she is assigned only "Facility 1 Transportation" domains. If she tries to view an asset in "Facility 2", she gets 403 Forbidden, because the asset belongs to a domain she does not have access to.

The two gates are orthogonal — independent. A user can:

  • Have permissions but no domains → can act, but sees nothing
  • Have domains but no permissions → sees data, but cannot act on it
  • Have both → full access within their scope
  • Have neither → no access
